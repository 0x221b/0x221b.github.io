Types of SQL

Before we get on to the exploitation a quick definition of SQL is needed. Skip this paragraph if you are happy with it already. SQL or Structured Query Language is used to retrieve information from databases. It comes in many flavours, the most common being MySQL and MSSQL and the syntax differs depending on the language used. but it basically boils down to sending a particular query to the database to retrieve the information you require. An oversimplified example would be SELECT * FROM tablename; that would return all the information from your table

Databases are used by websites to store all kinds of information, from product info, stock levels and even user credentials.

SQL Injection

Impact: 10        Risk: 10          Difficulty: 1-5 (depending on method and whether you use sqlmap)

So what is an SQL Injection attack?

An SQL Injection is pretty self explanatory it is a method of injecting malicious content into an SQL query through some method that allows you further access to the database than the creators intended. From this you can read, write and delete entries from databases compromising any data they have on there. This becomes especially dangerous if you can access the database with all the saved credentials. These should be stored as hashes (One would hope…) but these can then be saved to crack offline.

An SQL Injection can occur anywhere that user input is included, unsanitized, in an SQL query to the backend database. If a user can manuipulate the query then they can get it to return whatever they want from the database and even escalate access in some cases.

How do I find SQL Injection?

To find low hanging fruit I have compiled a gf json file to find commonly vulnerable params for SQL Injection.

Port Swiggers Web Academy suggests the following steps:

  1. Submitting the single quote character ‘ and looking for errors or other anomalies.
  2. Submitting some SQL-specific syntax that evaluates to the base (original) value of the entry point, and to a different value, and looking for systematic differences in the resulting application responses.
  3. Submitting Boolean conditions such as OR 1=1 and OR 1=2, and looking for differences in the application’s responses.
  4. Submitting payloads designed to trigger time delays when executed within an SQL query, and looking for differences in the time taken to respond.
  5. Submitting OAST payloads designed to trigger an out-of-band network interaction when executed within an SQL query, and monitoring for any resulting interactions.
